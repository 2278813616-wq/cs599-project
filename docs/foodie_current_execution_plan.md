# SuperFoodie 当前执行规划

本文记录当前阶段的落地顺序。长期方向见 `docs/foodie_future_planning.md`，本文只描述近期该怎么继续推进。

## 当前目标

先把“出去吃”链路做稳：

- 用户选择商圈和半径。
- 系统给出少量可信餐厅候选。
- 餐厅候选同时具备小红书种草信息和高德地点信息。
- 选中餐厅后，可以导出一份可用 PDF。
- 导出 PDF 后，把本次商圈、餐厅、饭后地点写入本地知识库，并给对应关系权重 `+1`。

暂时不追求复杂图谱可视化、多人系统、云端部署和完整 Obsidian 阅读体验。

## 推荐链路

### 1. 先查本地卡片库

当用户选择商圈后，系统先查本地已有卡片：

- 商圈卡片
- 餐厅卡片
- 小红书笔记卡片
- 饭后娱乐地点卡片
- 历史导出记录

如果本地已有足够可信候选，优先使用本地数据，减少实时搜索耗时。

### 2. 本地不足时联网补全

如果本地餐厅候选不足 3 个，再走联网补全：

- 小红书先搜探店笔记。
- 标题或正文能识别店名的笔记，进入候选。
- 识别到店名后，用高德按“店名 + 商圈中心坐标”反查。
- 高德匹配成功后，补齐楼层、地址、评分、图片、POI ID。

展示策略：

- 小红书可用时：最多 `2 个小红书种草卡片 + 1 个高德高分卡片`。
- 小红书不可用时：退化为 `3 个高德高分卡片`。

### 3. 小红书无店名的处理

主流程不应查询评论区。评论区接口风险较高，容易触发平台风控，近期默认关闭。

实时链路中：

- 只使用搜索结果和帖子详情页信息。
- 标题和正文抽不到店名的笔记不进入推荐。
- 仍找不到店名的笔记写入待处理队列或过滤缓存。
- 主流程继续返回高德候选。

后续如确实需要评论区补全，只能作为低频、手动触发或离线管理员任务：

- 不在用户主流程中自动触发。
- 必须有限速、缓存和失败黑名单。
- 找到疑似店名后，必须再用高德反查。
- 高德匹配成功才算解析成功。
- 匹配失败则进入待人工确认或后续重试。

笔记状态建议：

- `pending`：待后续解析。
- `resolved`：已找到店名，并已高德匹配成功。
- `unresolved_no_store_name`：搜索结果和帖子详情页都没有店名。
- `store_name_found_but_poi_missing`：找到店名，但高德匹配失败。
- `ignored`：纯菜谱、合集、广告、无地点价值。

以后实时搜索遇到同一个 note_id：

- `resolved`：直接使用缓存结果。
- `unresolved_no_store_name`：默认跳过。
- `ignored`：直接过滤。
- `pending`：不阻塞主流程，后续低频处理。

## 本地存储规划

当前建议先用本地结构化存储，不以 Obsidian Markdown 为核心。

推荐最小结构：

- `data/foodie_graph.json`：节点、边、权重。
- `data/cards/business_areas/`：商圈卡片。
- `data/cards/restaurants/`：餐厅卡片。
- `data/cards/entertainment/`：饭后娱乐地点卡片。
- `data/cards/xhs_notes/`：小红书笔记状态和解析结果。
- `data/assets/`：图片缓存和用户上传图片。

Obsidian Markdown 暂时只作为后续可读导出或管理员调试视图，不作为当前主要入口。

## 图谱节点

近期只需要这些节点：

- `business_area`：商圈。
- `restaurant`：餐厅。
- `entertainment`：电影院、电玩城、PS、台球、猫咖、展览等。
- `xhs_note`：小红书笔记。
- `pdf_export`：一次导出记录。

## 图谱边

近期只需要这些边：

- `restaurant -> belongs_to -> business_area`
- `entertainment -> near -> business_area`
- `xhs_note -> recommends -> restaurant`
- `pdf_export -> selected_restaurant -> restaurant`
- `pdf_export -> selected_area -> business_area`
- `pdf_export -> after_meal_place -> entertainment`

## 权重规则

当前只保留强信号：

- 导出 PDF：相关边权重 `+1`

暂时不把浏览、点击、展开详情、卡片选中计入权重，避免噪声污染用户画像。

## PDF 导出优先级

下一步优先修 PDF，因为它是当前最重要的强意图入口。

PDF 至少应该包含：

- 本次商圈。
- 出发点。
- 选中餐厅。
- 餐厅楼层和详细地址。
- 餐厅图片。
- 推荐菜。
- 小红书种草依据。
- 高德评分和人均。
- 饭后娱乐地点。
- 起点到餐厅、餐厅到饭后地点的路线说明。

导出完成后：

- 写入商圈卡片。
- 写入餐厅卡片。
- 写入饭后娱乐地点卡片。
- 写入小红书笔记卡片。
- 写入一次 PDF 导出记录。
- 给相关图谱边权重 `+1`。

## Milvus 的定位

Milvus 不负责结构化事实和关系。

结构化事实应由本地 JSON 或 SQLite 保存：

- 店名
- 地址
- 楼层
- 商圈
- 图片
- 评分
- 权重
- 导出次数

Milvus 只负责语义召回：

- 小红书笔记摘要
- 餐厅描述
- 用户反馈
- 商圈总结
- 饭后娱乐地点描述
- 历史推荐理由

后续适合用 Milvus 处理的问题：

- “找一个适合朋友聚餐、饭后还能逛的地方”
- “类似上次那个韩餐，但别太排队”
- “想吃辣但不要太油”
- “雨天适合吃完饭还能待一会儿的地方”

## 近期实施顺序

1. 修好 PDF 导出内容和版式。
2. 导出 PDF 后写入本地卡片和图谱边。
3. 建立小红书笔记状态缓存和失败过滤。
4. 暂缓评论区补全，只保留为未来可选的管理员低频任务。
5. 加商圈预热脚本，例如先跑 `武商梦时代`。
6. 页面增加本地卡片优先读取。
7. 后续再考虑 Milvus 语义召回和网页端档案页。

## 暂时不做

- 不做复杂知识图谱可视化。
- 不把 Obsidian 当主要阅读入口。
- 不把点击、浏览计入权重。
- 不做多人用户系统。
- 不做云端图片存储。
- 不强依赖 Milvus 完成当前推荐主链路。
