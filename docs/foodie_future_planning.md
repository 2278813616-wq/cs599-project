# SuperFoodie 未来规划备忘

本文记录侧聊中确认的长期方向，供主对话继续规划和实现时参考。当前只作为架构与产品设计依据，不代表已经实现。

## 产品定位

SuperFoodie 不应只是“搜索工具”，而应是“美食出行决策器”。

核心价值是替用户完成跨平台整合和决策：

- 小红书提供种草感、真实探店语境、推荐菜和评论区线索。
- 高德提供地点、地址、楼层、评分、距离和路线。
- 本地知识库沉淀商圈、店铺、娱乐地点、用户历史和反馈。
- 系统最终给用户少量可信候选，而不是更多搜索结果。

目标输出应是决策卡片：

- 去哪吃。
- 为什么适合。
- 店名、地址、楼层、评分。
- 推荐菜。
- 小红书种草依据。
- 饭后去哪玩或休息。
- 路线与导出 PDF。

## 本地存储方向

现阶段先采用本地存储，不急于多人化和云端化。

推荐优先使用 SQLite 作为核心结构化数据层，JSON 可用于调试、导出和备份。

原因：

- SQLite 比散乱 JSON 更接近未来数据库模型。
- 后续迁移到 PostgreSQL/MySQL 更自然。
- 适合存事实关系、权重、图片路径、用户行为。
- 本地单管理员使用足够简单。

Obsidian Markdown 暂时不要作为核心阅读入口。它可以作为后续导出格式、管理员调试视图或可读备份，但当前真正的查阅入口应该是网页端。

## 推荐数据表

未来可以围绕以下表设计：

- `business_areas`：商圈、商业体、坐标、半径、城市、别名。
- `places`：餐厅和休闲娱乐场所，包含地址、楼层、坐标、评分、类型、所属商圈。
- `notes`：小红书、微博等来源内容，包含笔记 ID、链接、摘要、作者、图片、关联店铺。
- `assets`：远程图片、本地缓存图、用户上传图。
- `edges`：节点关系和权重。
- `user_actions`：导出 PDF、选择方案等强信号行为。
- `user_feedback`：真实吃后反馈、上传图片、菜品评价、避雷信息。

## 知识图谱模型

节点类型：

- 商圈。
- 餐厅。
- 休闲娱乐场所。
- 小红书笔记。
- 菜品。
- 用户。
- 标签，例如川菜、朋友聚餐、排队、饭后逛、适合拍照。

边类型：

- 餐厅属于商圈。
- 娱乐地点属于商圈。
- 小红书笔记种草餐厅。
- 餐厅有推荐菜。
- 用户导出过某个商圈或店铺。
- 用户反馈过某个菜品。
- 餐厅附近可接饭后娱乐。

当前只采用低噪声权重规则：

- 导出 PDF：相关商圈、店铺、路线边权重 `+1`。

暂不把点击、浏览、展开详情纳入权重，避免误触污染用户画像。

## 图片策略

本地阶段支持三类图片：

- 远程图片 URL：小红书、高德返回图。轻量但可能过期或防盗链。
- 本地缓存图：下载到本地资产目录，适合长期沉淀。
- 用户上传图：用户真实反馈的核心资产，必须本地保存。

建议目录：

- `data/assets/restaurants/{place_id}/`
- `data/assets/dishes/{dish_id}/`
- `data/assets/notes/{note_id}/`

未来如果部署，可以把图片迁移到对象存储，例如 OSS、S3、R2。数据库中只保留图片 URL 或对象 key。

## Milvus 的定位

Milvus 不再承担主数据库职责。

它的定位是历史语义查询层，用于模糊召回：

- 小红书笔记摘要。
- 店铺描述。
- 用户真实反馈。
- 历史推荐理由。
- 商圈总结。
- 饭后娱乐场所描述。

适合回答：

- “找个类似上次那个韩餐的。”
- “我之前看过一家不太辣的川菜是哪家？”
- “有没有适合吃完饭还能逛的地方？”
- “按我以前喜欢的风格推荐一个商圈。”

流程上可以先由 Milvus 召回语义相关候选，再回到 SQLite/图谱中拿结构化事实、图片、权重和路线信息做最终排序。

## 小红书与高德的角色

小红书：

- 提供种草信息、真实体验、推荐菜、评论区店名线索。
- 不适合直接作为结构化事实源。
- 评论区店名可以做缓存：`note_id -> store_name`，避免每次慢查。

高德：

- 提供店名标准化、地址、楼层、评分、坐标、路线和部分图片。
- 适合做地点事实校验和补全。

两者应结合：

1. 小红书发现值得去的店。
2. 评论/正文抽店名。
3. 高德补地址、楼层、评分和位置。
4. 写入本地知识库。

## 商业化与合规边界

技术上，本地 SQLite/JSON 迁移到服务端数据库是可行的。

如果未来多人使用或商业化，需要注意：

- 店名、地址、评分等事实信息通常版权风险较低，但仍受平台 API 条款约束。
- 小红书笔记正文、图片、用户评论有版权和平台规则风险，不应大量公开复制或二次分发。
- 高德 POI 和图片也受高德 API 协议约束，不能随意长期公开缓存或绕开授权。
- 用户上传图片和反馈可以成为核心资产，但需要用户授权协议。

商业化时应出售“整合、推荐、记忆、规划能力”，而不是出售第三方平台内容本身。

## 阶段建议

第一阶段：设计本地结构化模型。

- SQLite 表结构。
- 本地图片缓存策略。
- 节点和边定义。
- 导出 PDF 后权重 `+1`。
- Milvus 只做语义历史召回。

第二阶段：把卡片落盘。

- 商圈卡片。
- 餐厅卡片。
- 娱乐地点卡片。
- 小红书笔记卡片。
- 高德补全信息。

第三阶段：反馈闭环。

- 用户上传真实图片。
- 用户填写吃后反馈。
- 反馈进入结构化库和 Milvus。

第四阶段：网页端知识库。

- 商圈档案页。
- 店铺档案页。
- 我的去过记录。
- 种草来源页。
- 饭后娱乐推荐页。

第五阶段：多人和商业化。

- 从 SQLite 迁移到 PostgreSQL。
- 图片迁移对象存储。
- 用户权限、授权协议和合规边界。
