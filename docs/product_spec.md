# Product Specification: SuperFoodie 超级吃货智能助手

## 1. 产品愿景与背景

传统的美食推荐平台大多零散：菜谱推荐不考虑用户家里的调料平替或物理火候技巧；餐厅推荐不考虑往期去重和用户的临时身体状况（如感冒咳嗽、痛风等），且缺乏吃喝玩乐一站式路线规划。

**SuperFoodie** 的目标是做吃货最贴心的决策大脑：
*   **家庭烹饪**：提供极度细致的火候指导，并自动拦截不合适食材。
*   **外出就餐**：不仅主餐推荐，还有饮料甜品和餐后娱乐推荐，一键计算出行交通时间成本，生成图文 PDF 决策报告。
*   **核心保障**：底层依托 Obsidian 知识图谱（Graph-RAG）与 Milvus 长期记忆库，确保决策有理有据。

## 2. 核心功能规范

### 2.1 自己做 (Home Chef)
*   **【功能 2.1.1】多样化菜单推荐**：
    *   用户可指定就餐人数（单人/多人）和目标菜系/口味偏好。
    *   系统基于 `food_search` 模块，结合本季热门菜，推荐一套包含多道菜品的可行菜单，并附带食材及调料的“补齐清单”。
*   **【功能 2.1.2】精细化物态烹饪指导**：
    *   做菜步骤中，必须包含明确的操作物理判定标志（如：油温判断法、食材状态物理特征）。
    *   支持用户中途追问。若用户在某一步骤表示某种调料/食材缺失，Agent 多步骤推理并返回最佳平替食材或步骤修正方案。
*   **【功能 2.1.3】去腥与新鮮度分支处理**：
    *   做肉类/海鲜等菜肴时，Agent 会前置提示并询问用户食材的新鲜程度（如“新鲜”还是“冷冻/隔夜”）。
    *   若是新鲜食材，步骤中省去繁琐去腥以体现原味；若是不新鲜，自动插入料酒、葱姜水、焯水等物理/化学去腥操作步骤。

### 2.2 出去吃 (Gourmet Explorer)
*   **【功能 2.2.1】吃喝玩乐一条龙推荐**：
    *   支持单人/多人就餐模式。在用户提出找店诉求后，系统推荐主餐餐厅（并提取其必吃特色招牌），同时搭配推荐半径 1 公里内的热度甜品/茶饮（如喜茶、蜜雪冰城）及餐后娱乐（台球、影院等）。
*   **【功能 2.2.2】高德地图耗时计算**：
    *   用户提供就餐起点与出行方式（如：打车、地铁、步行）。
    *   系统根据地图路线 API 计算到各推荐地点的出行耗时与路线，并呈现在探店计划中。
*   **【功能 2.2.3】图文 PDF 吃货指南导出**：
    *   在选店结束后，用户可一键导出吃货指南 PDF。PDF 排版精致，包含各推荐餐厅的比对表、行程路线时间线以及健康安全分析图谱。

### 2.3 饮食健康专家干预 (Health & Medical Diet Guard)
*   **【功能 2.3.1】Graph-RAG 膳食安全网**：
    *   从长期记忆中自动调取用户的身体指标、忌口（如不吃香菜）、过敏源（如花生过敏）及临时身体异常（如感冒咳嗽、痛风、糖尿病）。
    *   由 `HealthExpertAgent` 结合 Obsidian 本地双链知识库，对所有推荐的菜谱或餐厅菜品执行拦截。若有违禁食材（如痛风患者推荐了海鲜，感冒患者推荐了辛辣），自动予以拦截，发出红色警示，并在决策中说明基于图谱的拦截原因（如：“因为洋葱在图谱中标记为[[辛辣发物]]，而您正处于[[感冒]]阶段，故拦截”）。

### 2.4 记忆去重与穿透 (Memory Deduplication)
*   **【功能 2.4.1】近期就餐足迹去重**：
    *   所有已食用的菜品和去过的餐厅均被写入 Milvus 长期就餐表。
    *   系统在进行“今天吃什么”推荐时，自动查阅最近 7 天的足迹，在前置环节过滤这些已吃菜品，保证吃货的新鲜感。
*   **【功能 2.4.2】意图主动穿透机制**：
    *   如果用户显式指出诉求（如：“我今天就是想吃昨天那家火锅”），意图解析将绕过去重器，直接透传该推荐。

## 3. 质量要求与 Harness 评估阈值

为确保大作业满足“生产级水平”并获取加分项，系统将执行一系列 Harness 约束测试（通过 `pytest` 和自定义的自动化 Benchmark 进行监控）：

*   **API 密钥安全性**：Harness 静态分析工具对工程进行 100% 检索，凡发现代码中包含明文 API Key 样式硬编码者判定为 Failed。
*   **方差稳定性**：对同一个就餐诉求在相同环境下独立请求 3 次，获得的推荐菜单/餐厅匹配度方差必须小于 10%。
*   **健康拦截成功率**：在输入过敏或疾病标签时，不合规食材的拦截通过率必须为 100%。
*   **审计日志持久化率**：每次决策的所有中间推理（尤其是 Graph-RAG 的拦截路径、高德出行耗时）100% 实时写入本地追加写文件 `logs/audit.jsonl` 中。
*   **交互时效性**：每一轮 Agent 对话和决策平均响应时间必须在 10 秒之内（如遇超时，Harness 报错）。
